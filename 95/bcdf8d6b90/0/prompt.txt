### 1a. Full Vertical Flow Tests (the key gap)

Test the complete path: entity service call → lock provider sync → coordinator
update → WS subscription delivers correct data.

| Test | Action | Verify |
|------|--------|--------|
| `test_pin_set_via_service_reflects_in_subscribe_code_slot` | Subscribe slot 1, set PIN text entity to "9999" via `text.set_value` | WS update has `pin == "9999"`, lock data has matching code |
| `test_pin_clear_via_service_reflects_in_subscribe_code_slot` | Subscribe slot 1, clear PIN text entity | WS update has `pin is None` |

| `test_set_lock_usercode_reflects_in_subscribe_lock_codes` | Subscribe lock 1, call `set_lock_usercode` for unmanaged slot 3 | WS update shows slot 3 with code and `managed: False` |
| `test_update_slot_condition_reflects_in_subscribe_code_slot` | Subscribe slot, add condition via `update_slot_condition`, set condition entity ON | WS update includes condition data |

field renames, missing fields, and type mismatches.

| Test | WS Command | Assert |

| `test_subscribe_lock_codes_response_shape` | `subscribe_lock_codes` | All fields from `LockCoordinatorData` present |
| `test_subscribe_lock_codes_masked_shape_contract` | `subscribe_lock_codes` (reveal=False) | `code is None`, `code_length` is int |

### Implementation Pattern

```python
# Full flow test pattern:
async def test_pin_set_via_service_reflects_in_subscribe_code_slot(
    hass, mock_lock_config_entry, lock_code_manager_config_entry, hass_ws_client
):
    ws_client = await hass_ws_client(hass)
    # Subscribe
    await ws_client.send_json({...subscribe_code_slot...})
    msg = await ws_client.receive_json()  # success
    initial = await ws_client.receive_json()  # initial event
    assert initial["event"]["pin"] == "1234"

    # Set PIN via service (triggers sync → coordinator → WS update)
    await hass.services.async_call("text", "set_value", {...}, blocking=True)
    await hass.async_block_till_done()

    # Force coordinator refresh to reflect lock state
    lock = lock_code_manager_config_entry.runtime_data.locks[LOCK_1_ENTITY_ID]
    await lock.coordinator.async_refresh()
    await hass.async_block_till_done()

    update = await ws_client.receive_json()
    assert update["event"]["pin"] == "9999"
```

<summary>Agent "Fix PIN clear requires disable first" completed</summary>
<result>Here is a summary of the investigation and fix:

**Investigation findings:**

The PIN validation in `text.py` (`async_set_value`) rejected empty values on enabled slots by creating a persistent notification and returning early. The switch entity (`switch.py`) separately prevented enabling a slot without a PIN. Together, these enforced the invariant: no enabled slot without a PIN. However, the rejection was silent from the frontend's perspective -- the service call succeeded but the value didn't change, with only an easily-missed persistent notification as feedback.

**Decision: Bug (UX issue)**

The safety invariant is valid, but the implementation was user-hostile. The correct behavior is to auto-disable the slot when the PIN is cleared, rather than silently rejecting the action.

**Changes made:**

1. `/Users/raman/projects/lock_code_manager/.claude/worktrees/agent-afde91c1/custom_components/lock_code_manager/text.py` -- Changed `async_set_value` to call `switch.turn_off` on the enabled switch when clearing a PIN on an enabled slot, instead of rejecting with a persistent notification. 2. `/Users/raman/projects/lock_code_manager/.claude/worktrees/agent-afde91c1/tests/test_text.py` -- Updated the test to verify that clearing a PIN on an enabled slot results in both the PIN being cleared and the slot being disabled.
