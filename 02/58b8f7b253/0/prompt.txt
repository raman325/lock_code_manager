Live Matter Lock Testing — Lock Code Manager

  We're testing the Lock Code Manager (LCM) custom integration against a real Matter lock: lock.raman_office (10 slots, currently all empty).

  Architecture context (from codebase analysis):
  - Matter provider reports codes as SlotCode.UNKNOWN (write-only — PINs can't be read back) or SlotCode.EMPTY
  - LCM uses SlotSyncManager (one per lock×slot) with 5-second tick intervals to reconcile desired vs actual state
  - In-sync for UNKNOWN codes compares against _last_set_pin (the PIN we last sent)
  - Push events from Matter DoorLock cluster (event 4 = LockUserChange) drive real-time updates
  - Circuit breaker disables a slot after 3 failed sync attempts in 5 minutes
  - Duplicate PINs raise DuplicateCodeError → slot disabled + repair issue created
  - Config flow clears existing codes on selected slots before managing them

  What you (the agent) can do:
  - Call HA services via MCP tools (ha_call_service, ha_get_state, ha_search_entities, ha_get_device, etc.)
  - Read Matter lock state: matter.get_lock_users (service call with return_response: true)
  - Set codes directly: matter.set_lock_usercode with {slot, pin_code}
  - Clear codes directly: matter.clear_lock_usercode with {slot}
  - Read LCM entity states (sensors, binary_sensors, switches, text entities)
  - Call LCM websocket commands via ha_call_service if needed

  What the user must do manually:
  - Create/modify LCM config entries (via HA UI config flow)
  - Enable/disable slots (via HA UI or entity service calls — confirm with user first)
  - Restart integrations if needed
  - Physical keypad operations (Phase 8)

  Test Phases (execute in this order):

  Phase 1 — Baseline (DONE)

  Lock found. 10 slots, all empty. matter.get_lock_users returns {max_users: 10, users: []}.

  Phase 2 — Direct set/get round-trip

  1. Call matter.set_lock_usercode on slot 2 with pin 1234
  2. Call matter.get_lock_users to confirm slot 2 is populated
  3. Expected: Slot 2 shows as occupied (PIN won't be readable, just presence)

  Phase 3 — Duplicate code detection

  1. Call matter.set_lock_usercode on slot 3 with the SAME pin 1234
  2. Expected: Either the lock rejects it (error) or accepts it (Matter spec allows duplicates on some locks). Record the behavior — LCM's provider catches duplicates at the application layer, but the lock itself may or may not enforce uniqueness.

  Phase 4 — LCM config entry setup

  1. Ask user to create an LCM config entry via HA UI:
    - Select lock.raman_office
    - Configure slots 1-3 (slot 2 already has a code from Phase 2)
    - Slot 1: name="Test User 1", PIN="5678", enabled=true
    - Slot 2: name="Test User 2", PIN="1234", enabled=true
    - Slot 3: name="Test User 3", PIN="9012", enabled=true
  2. Expected during config flow: LCM should detect existing code on slot 2 (and slot 3 if Phase 3 succeeded) and ask to clear them before proceeding
  3. After setup, wait ~10 seconds for sync, then:
    - Read all LCM entities: sensor.code_*, binary_sensor.active_*, binary_sensor.in_sync_*, text.pin_*
    - Call matter.get_lock_users to verify codes were set on the lock
  4. Expected: All 3 slots in-sync, codes set on lock (showing as UNKNOWN)

  Phase 5 — External drift detection

  1. Directly change slot 1's code: matter.set_lock_usercode slot 1 pin 9999 (bypassing LCM)
  2. Wait for LCM's hard refresh interval (check lock.hard_refresh_interval — likely 5 minutes, or trigger manually)
  3. Read binary_sensor.in_sync_* entities for slot 1
  4. Expected: Slot 1 goes out-of-sync, then LCM re-syncs it back to "5678" on next tick
  5. Verify with matter.get_lock_users that the lock has the correct code again

  Phase 6 — Slot disable/enable cycle

  1. Ask user (or call service) to disable slot 1 (set the enabled switch to OFF)
  2. Wait ~10 seconds
  3. Read matter.get_lock_users — slot 1 should be cleared
  4. Re-enable slot 1
  5. Wait ~10 seconds
  6. Read matter.get_lock_users — slot 1 should be set again
  7. Expected: Clean disable → clear → re-enable → set cycle with in-sync throughout

  Phase 7 — Duplicate PIN via LCM

  1. Ask user to change slot 3's PIN to "5678" (same as slot 1) via the text entity or UI
  2. Expected: LCM detects duplicate, disables slot 3, creates repair issue
  3. Verify: slot 3 disabled, repair issue visible, slot 3's code cleared on lock

  Phase 8 — Clear and cleanup

  1. Ask user to delete the LCM config entry
  2. Expected: All managed slots cleared on the lock
  3. Verify with matter.get_lock_users — all slots empty again

  Phase 9 — Physical keypad (optional, requires user action)

  1. Set up a fresh config entry with slot 1, PIN "1111"
  2. Ask user to use the physical keypad to unlock with PIN "1111"
  3. Check HA logs/events for LockOperation event (event 2) from Matter
  4. Expected: Event fires with correct credential slot attribution
