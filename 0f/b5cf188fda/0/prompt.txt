## Background

A Home Assistant custom integration called Lock Code Manager (LCM) recently
migrated from calling the legacy `zwave_js.set_lock_usercode` service to using
`homeassistant.components.zwave_js.lock_helpers.async_set_credential`. This is
the new unified user/credential API HA added in 2026.6.x that exposes the
unified `node.access_control` surface from node-zwave-js v15.23.4+.

After LCM 4.0.x ships this change, some users see their existing Z-Wave locks
break with this exact error per slot:

    "Credential slot for pin_code must be between 1 and 0."

Two important data points from the issue thread:

- The bug affects SOME users but not all. At least one user (the LCM
  maintainer himself) with a 700-series User Code CC ("UC")-only lock keeps
  working fine.
- jhenkens confirmed they use "ZWaveJS UI" but hasn't yet shared lock
  make/model. randallkn is also affected, no lock details.

PR #1252 (open, not merged) proposes a local mitigation that (1) detects
UC-only locks via a fallback walk of UserCodeCC values and (2) bypasses
`lock_helpers.async_set_credential` entirely by calling
`node.access_control.set_credential` directly. That unblocks affected users
without waiting for an upstream fix. We now want to land the actual root-cause
fix upstream — both to ship a cleaner LCM and so other integrations using
`lock_helpers` don't hit the same wall.

## The full call chain (verified by tracing the local checkouts before this prompt was written — confirm against the fresh fetches)

    LCM → homeassistant.components.zwave_js.lock_helpers.async_set_credential
        → validates `1 <= credential_slot <= type_cap.number_of_credential_slots`
          and raises ServiceValidationError("credential_slot_out_of_range") on

        → "access_control.get_credential_capabilities_cached" WS RPC

## Working hypotheses (need to be verified or refuted)

These are POSSIBILITIES. Do not assume any of them; treat them as a starting
set and report which (if any) the evidence actually supports.

### HYPOTHESIS A — Dispatch bug per the FIXME

Some locks advertise BOTH "User Code" CC (0x63) and "User Credential" CC
(0x83). For those, `#usesUserCredentialCC` returns true → U3C branch runs.
If the U3C interview is incomplete (or the lock advertises U3C but the U3C
surface is empty / not fully implemented), the U3C branch returns either an
empty `supportedCredentialTypes` map (would yield HA's
`credential_type_not_supported` error — DIFFERENT message) OR a map with PIN
present but `numberOfCredentialSlots: 0` (would yield "between 1 and 0"). The
latter is narrow but possible.

### HYPOTHESIS B — Value DB miss in UC branch

Lock advertises UC only. UC branch runs. But `UserCodeCCValues.supportedUsers`
is `undefined` in the cached value DB (interview never completed, was
interrupted, or got purged on a driver upgrade), so `?? 0` returns 0 →
`numberOfCredentialSlots: 0` → "between 1 and 0".

### HYPOTHESIS C — HA wrapper bug (independent of driver)

`lock_helpers.async_set_credential` validates against the cached cap BEFORE
calling the driver. The driver itself has its own range check on the actual
`setCredential` call. The HA pre-validation is over-strict and produces a
misleading error message when the cache is stale (intermediate interview
state). Even if the driver is right, HA's wrapper rejects the call.

These are not mutually exclusive; A or B explain why the value is 0, C
explains why the user-visible failure mode is so unfriendly.
