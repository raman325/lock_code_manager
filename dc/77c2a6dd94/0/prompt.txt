# Plan: Duplicate Code Detection + Architecture Documentation

## Context

Users report infinite sync loops when a lock rejects a PIN that duplicates another slot's code. PR #899 added reactive handling via Z-Wave event 15, but some locks don't emit this event reliably. Since the coordinator already stores ALL slots (managed + unmanaged), we can detect duplicates pre-flight before sending to the lock.

Additionally, the codebase lacks documentation on the data flow between coordinator, provider, and sync sensor, and the managed/unmanaged distinction is implicit.

## Key Design Decisions

- **DuplicateCodeError (exception B approach):** Provider raises, callers decide policy. Sync path disables + notifies. Direct path propagates error.
- **Masked slots:** Can't detect duplicates against `****` values. Event 15 handler stays as safety net.
- **Architecture doc over module docstrings:** Cross-cutting data flow belongs in one place, not split across 4 files.
