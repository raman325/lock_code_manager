  We're testing the Lock Code Manager (LCM) custom integration against a real Matter lock: lock.raman_office (10 slots, currently all empty).

  Architecture context (from codebase analysis):
  - Matter provider reports codes as SlotCode.UNKNOWN (write-only — PINs can't be read back) or SlotCode.EMPTY
  - LCM uses SlotSyncManager (one per lock×slot) with 5-second tick intervals to reconcile desired vs actual state
  - In-sync for UNKNOWN codes compares against _last_set_pin (the PIN we last sent)
  - Push events from Matter DoorLock cluster (event 4 = LockUserChange) drive real-time updates
  - Circuit breaker disables a slot after 3 failed sync attempts in 5 minutes
  - Duplicate PINs raise DuplicateCodeError → slot disabled + repair issue created
  - Config flow clears existing codes on selected slots before managing them

## Report: LCM Z-Wave JS Test Fixtures vs HA Z-Wave JS Test Fixtures

### 1. How LCM fixtures work today

The LCM provider tests in `tests/providers/conftest.py` closely mirror HA's pattern:

- **`zwave_client`** fixture patches `homeassistant.components.zwave_js.ZwaveClient`, creates a mock client with connect/listen/disconnect side effects, builds a real `Driver` and controller `Node` from JSON fixture files in `tests/providers/fixtures/zwave_js/`.
- **`lock_schlage_be469`** creates a real `Node` object from fixture JSON and registers it on the client's driver controller.
- **`zwave_integration`** creates a `MockConfigEntry` for `zwave_js`, calls `async_setup`, yielding a fully loaded Z-Wave JS config entry.
- **`lock_entity`** finds the lock entity registry entry created by the integration setup.
- **`zwave_js_lock`** (in `test_zwave_js.py`) constructs a `ZWaveJSLock` provider instance directly, passing the registries and lock entity -- it does NOT go through LCM's own config entry setup for most tests.

Most tests then call `async_setup_internal` with a throwaway `MockConfigEntry` and patch methods like `async_call_service`, `_get_usercodes_from_cache`, and `code_slot_in_use` to avoid hitting real Z-Wave commands.

### 2. How HA Z-Wave JS tests work

HA's conftest (`tests/components/zwave_js/conftest.py`) is structurally identical in the client/driver/node layer. The key differences:

- **`client` fixture** (named `client`, LCM uses `zwave_client`) -- same pattern, same `connect/listen/disconnect` mocks, same `async_send_command` side effect.
- **`integration` fixture** takes a `platforms` fixture parameter, allowing tests to restrict which platforms load. This keeps tests fast and focused.
- **Node fixtures** follow the exact same `Node(client, copy.deepcopy(state))` + register on controller pattern.
- HA's node fixture JSON files live in `tests/components/zwave_js/fixtures/` and are loaded via `load_json_object_fixture` (HA test helper). LCM loads them via a local `load_json_fixture` function from `tests/providers/fixtures/zwave_js/`.

### 3. Gaps for proper end-to-end test coverage

1. **Provider constructed manually, not via LCM entry setup.** Most tests build `ZWaveJSLock` directly rather than going through `hass.config_entries.async_setup` with a real LCM config entry. The later `test_duplicate_code_*` tests do use `_setup_lcm_entry` for full setup -- this pattern should be the default.
2. **Heavy patching of provider internals.** `async_call_service`, `_get_usercodes_from_cache`, `code_slot_in_use` are frequently patched. True end-to-end tests should let calls flow through to the mock client's `async_send_command`.
3. **`mock_zwave_usercodes` is only used for duplicate-code tests.** This fixture (wrapping `get_usercodes`/`get_usercode` with mutable state backed by `async_send_command`) is the right pattern for end-to-end but is not used by the majority of tests.
4. **No coordinator lifecycle testing.** Most tests assign `zwave_js_lock.coordinator = MagicMock()` instead of letting the real coordinator run.
5. **Fixture JSON files are copies.** LCM maintains its own copies of `lock_schlage_be469_state.json` rather than referencing HA's.

### 4. Rough refactor plan

1. **Make `_setup_lcm_entry` (or equivalent) the default test helper** -- all tests should create a real LCM config entry and let the integration set up the provider, coordinator, and entities.
2. **Expand `mock_zwave_usercodes` to all tests** -- wire `get_usercodes`/`get_usercode` through mutable state so `async_get_usercodes`, `async_set_usercode`, and `async_clear_usercode` exercise real code paths end-to-end.
3. **Remove manual `ZWaveJSLock` construction** -- retrieve the provider instance from `lcm_entry.runtime_data.locks[entity_id]` instead.
4. **Stop patching `async_call_service`** -- let it call through to HA's service layer backed by the mock Z-Wave client.
5. **Let the real coordinator run** -- remove `zwave_js_lock.coordinator = MagicMock()` assignments; use `async_trigger_sync_tick` helpers for controlled refresh.</result>

## Test Refactoring Plan

### 1. Current Test Infrastructure

**`tests/conftest.py`** provides four key autouse fixtures plus two entry-level fixtures:
- `auto_enable_custom_integrations` -- enables custom integration loading
- `auto_setup_mock_lock` -- patches `INTEGRATIONS_CLASS_MAP` to include `MockLCMLock` for the `"test"` domain
- `disable_rate_limiting` -- zeroes out `_min_operation_delay` on all `BaseLock` instances
- `mock_lock_config_entry` -- creates a **fake "test" integration** with mock lock + calendar entities, sets up its config entry, yields it, and unloads on teardown. This provides the prerequisite lock/calendar entities that Lock Code Manager needs.
- `lock_code_manager_config_entry` -- creates a real LCM config entry using `BASE_CONFIG` (2 locks, 2 slots), calls `async_setup_entry` through the real code path, yields the entry, unloads on teardown.

**`tests/common.py`** defines `MockLCMLock` (a `BaseLock` subclass with in-memory codes), `MockLockEntity`, `MockCalendarEntity`, entity ID constants, and `BASE_CONFIG`.

**Key finding:** Most tests already go through real `async_setup_entry`. The `mock_lock_config_entry` fixture is not mocking LCM -- it creates the prerequisite lock entities that LCM manages. Only `test_config_flow.py` mocks out `async_setup_entry`/`async_unload_entry` (intentionally, since config flow tests only need the flow logic).
