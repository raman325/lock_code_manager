### Critical (90-100)

**1. `setup_push_subscription` violates the synchronous `@callback` contract -- race condition and broken idempotency** (Confidence: 95)

File: `custom_components/lock_code_manager/providers/zigbee2mqtt.py`, lines ~326-371

The base class defines `setup_push_subscription` as `@callback` (synchronous, must not await). Both Z-Wave JS and Matter implement it fully synchronously -- subscription happens inline and the unsub handle is stored before return.

The Zigbee2MQTT implementation does `hass.async_create_task(_async_subscribe())` and returns immediately. This means:

1. `_unsubscribe` is still `None` when the method returns, so the idempotency guard (`if self._unsubscribe is not None: return`) is ineffective. A rapid reconnect cycle (which calls `subscribe_push_updates` again) will spawn duplicate subscriptions.
2. The caller (`subscribe_push_updates` in `_base.py`) expects that on successful return, the subscription IS active. Instead, it might silently fail later inside the task with no error propagation.
3. If the task fails, the `LockDisconnected` exception is swallowed (only logged) and the system believes push is active.

**Fix:** HA's `async_subscribe` for MQTT needs to be awaited, but `setup_push_subscription` is `@callback`. The pattern used by other providers is that the subscription API itself is synchronous. Since MQTT's `async_subscribe` is async, you need a different approach. Options:
- Override `async_setup()` to do the initial subscription (await `async_subscribe` there and store the unsub). Then make `setup_push_subscription` a no-op that just validates `_unsubscribe is not None` or re-subscribes synchronously if you can cache a reference.
- Or, coordinate via a flag: set a `_subscribing` sentinel immediately so the idempotency guard works, then schedule the task. On failure, clear the sentinel.

**2. `call_soon_threadsafe` is unnecessary and introduces an extra scheduling hop** (Confidence: 82)

File: `custom_components/lock_code_manager/providers/zigbee2mqtt.py`, lines ~345-355

The code wraps the handler in `hass.loop.call_soon_threadsafe(_deliver)`, which is only needed if invoked from a non-event-loop thread. **Fix:** Remove the `call_soon_threadsafe` wrapper and call `self._process_z2m_device_payload(payload)` directly in `message_received`. Alternatively, if you want to be defensive about thread safety, verify that HA's MQTT guarantees callbacks on the event loop (it does since HA 2023.x+).

## Issue 1: Timeout treated as "slot empty" instead of signaling failure

**Location**: `/custom_components/lock_code_manager/providers/zigbee2mqtt.py` lines 521-529 (from diff)

**Status: NOT FIXED**

The `async_get_usercodes` method still treats a timeout as `SlotCode.EMPTY`:

**Severity**: HIGH

**Issue Description**: When the lock firmware is slow, the broker is congested, or the device is temporarily unreachable, the 10-second timeout fires and the slot is recorded as `SlotCode.EMPTY`. The sync manager will then see "empty slot" and attempt to clear or skip that slot -- treating a communication failure as authoritative state. The comment in the diff at line 504 acknowledges this was a deliberate design choice ("Parallel gathers plus per-slot timeouts used to raise and fail the entire refresh, leaving coordinator.data empty"), but the fix swings too far in the other direction.

**Analysis:** This bare `except Exception` block around `asyncio.wait_for(future, timeout=10.0)` was already present before this PR. The PR correctly elevates it from `LOGGER.debug` to `LOGGER.warning`, which is appropriate since an unexpected exception during PIN retrieval should be visible in logs without enabling debug mode.

The broad catch here is **justified by context**: the future could be resolved by a callback that processes MQTT messages, and any unexpected exception during that resolution (e.g., `CancelledError` from HA shutdown, malformed data in the callback) should not crash the entire usercode refresh. ## Issue 1: `_parse_pin_response` silently treats unrecognized response shapes as "slot empty"

**Location:** `/Users/raman/projects/lock_code_manager/custom_components/lock_code_manager/providers/zha.py`, line 338

**Severity:** HIGH

**Issue Description:** When the response from `get_pin_code` matches neither the attribute-based format (`hasattr(result, "user_status")`) nor the tuple/list format (`len(result) &gt;= 4`), the method silently returns `(UserStatus.Available, "")`. The caller at line 242-245 interprets this as `SlotCode.EMPTY`. This means if zigpy changes its response format, or the lock returns an error response object that has a different shape, existing PIN codes will appear to be empty. The sync engine will then either try to re-program the code (wasted churn) or, worse, if the slot is supposed to be cleared, it will think it already is.

The raw result IS logged at debug level (line 235-239), but there is no indication that parsing fell through to the default branch. Someone debugging "why did the sync engine think slot 3 was empty when it has a code?" would need to know to look at the raw debug output and manually reason about the parse logic.

**User Impact:** A code that exists on the lock could be reported as empty, causing unnecessary sync operations or masking the fact that the lock's firmware responded in an unexpected way.

**Recommendation:** Log a warning when the response falls through to the default return, so there is a clear signal that something unexpected happened:

```python
    @staticmethod
    def _parse_pin_response(result: Any) -&gt; tuple[int, str]:
        """Extract (user_status, pin_code) from a get_pin_code response."""
        if hasattr(result, "user_status"):
            pin = getattr(result, "code", "") or ""
            if isinstance(pin, bytes):
                pin = pin.decode("utf-8", errors="ignore")
            return result.user_status, str(pin)
        if isinstance(result, (list, tuple)) and len(result) &gt;= 4:
            pin = result[3]
            if isinstance(pin, bytes):
                pin = pin.decode("utf-8", errors="ignore")
            return result[1], str(pin) if pin else ""
        _LOGGER.warning(
            "Unrecognized get_pin_code response format (type=%s), "
            "treating slot as empty: %s",
            type(result).__name__,
            result,
        )
        return DoorLock.UserStatus.Available, ""
```

Note: since `_parse_pin_response` is a `@staticmethod`, the lock entity_id is not available here. The warning is still useful because the caller already logs the entity_id and slot number at debug level on line 235-239, providing the surrounding context needed to correlate.
