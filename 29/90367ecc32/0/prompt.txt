**Confidence: 83**

**File:** `/Users/raman/projects/lock_code_manager/custom_components/lock_code_manager/sync.py`, lines 373–376 and lines 482–490


The post-sync re-check at lines 447–453 re-resolves state after the `async_refresh` completes. This works correctly for the in-flight sync path.

The gap is the following sequence:
1. Sync is in progress (lock held), coordinator update arrives → task is suppressed (line 484).
2. Sync completes, `_perform_sync` returns True.
3. `coordinator.async_refresh()` is called at line 433 — this fires coordinator listeners again. Since the lock is still held during the `await`, the new listener callback creates a task but the lock-check in `_async_check_and_sync` discards it.
4. The post-sync re-check at line 448 runs `_resolve_slot_state(None)` which reads the freshly-updated coordinator data directly. This re-check does catch the updated state.

```python
if not self._coordinator.last_update_success and not self._retry.active:
    return None
```

Since `self._retry.active` is False at that point (the retry was just scheduled, not running), this returns None even though a retry is queued. The slot will stay `in_sync=False` until the retry fires. This is correct behavior but the retry guard in `_resolve_slot_state` conflates "retry is scheduled" (`pending`) with "retry is executing" (`active`). The `OneShotRetry.active` property is only True during callback execution (see `util.py` line 146). The condition should be `not self._retry.pending` to cover the waiting-for-retry state, or `not (self._retry.pending or self._retry.active)`:

```python
if not self._coordinator.last_update_success and not self._retry.pending:
```

**Confidence: 88** — The `OneShotRetry.active` property is explicitly documented as True only during execution. Using it instead of `pending` here is a logic error that will cause updates to be dropped during the retry window.

### Issue 1: `_aio_lock.locked()` check in `_handle_coordinator_update` is not a reliable guard (Confidence: 88)

**File:** `/Users/raman/projects/lock_code_manager/custom_components/lock_code_manager/sync.py`, lines 506–516

In `_handle_coordinator_update`, the method checks `self._aio_lock.locked()` and returns early if locked. But because `_handle_coordinator_update` is a `@callback` (synchronous, runs on the event loop), and `async_create_task` just schedules a coroutine — it does not hold the lock at the time of the check. The lock is only held while `_async_check_and_sync` is actually running inside its `async with self._aio_lock` block.

The problem: if `_async_check_and_sync` is awaiting `_perform_sync` or `async_refresh()` (both of which are long-running awaits), the event loop can run other callbacks, including `_handle_coordinator_update`. At that point, `_aio_lock.locked()` returns `True` and `_handle_coordinator_update` returns early — dropping the coordinator update entirely without scheduling any retry. If `_async_check_and_sync` holds the lock through a coordinator refresh, and the coordinator fires an update while waiting, that update is silently dropped even though post-sync re-evaluation (`_resolve_slot_state(None)`) might now read stale state (before the fired update is reflected in `_coordinator.data`).

This is an inherent design tension: the coordinator fires `_handle_coordinator_update` precisely after refreshing its data, but if the lock is held during `await self._coordinator.async_refresh()`, the notification is dropped. The post-sync re-read at line 462 uses `_resolve_slot_state(None)` which reads `self._coordinator.data` directly — so it does capture the fresh data. This means the logic is actually correct in the common case: the refresh result is available immediately after `async_refresh()` returns, and the re-read at line 462 picks it up.

If the lock is held during the initial `_async_check_and_sync()` call in `async_start`, and a coordinator update arrives, the early return at line 506 drops it. **File:** `/Users/raman/projects/lock_code_manager/custom_components/lock_code_manager/sync.py`, lines 361–367

This catch-all is meant for unexpected errors, but if the error is actually permanent (e.g., a provider bug that raises the same exception on every call), the tick loop will retry forever at 5-second intervals: `_dirty=True` → tick fires → `_perform_sync` → exception → `_dirty=True` → tick fires → repeat.

### ISSUE 4: `async_stop` does not guard against a concurrently running `_async_tick`

**Location:** `/custom_components/lock_code_manager/sync.py`, `async_stop` method

**Severity:** MEDIUM

**Issue Description:**
`async_stop` is a synchronous `def` (not async), so it runs atomically within a single event loop iteration. It sets `self._started = False`, cancels the tick unsub, cleans up listeners, and resets the tracker. However, if `_async_tick` is currently mid-execution (sitting at an `await` point like `_perform_sync` or `coordinator.async_refresh()`), the tick will resume after that await and continue operating on a stopped manager.

Specifically, after `async_stop`:
- `_async_tick` could call `self._write_state()` on an entity that has been removed
- `_async_tick` could call `self._coordinator.async_refresh()` after the coordinator unsub
- `_async_tick` could set `self._dirty = True`, though this is harmless

The `_tick_unsub()` cancels future scheduled ticks, but does NOT cancel an already-running `_async_tick` coroutine.

**User Impact:** Could cause `async_write_ha_state` on a removed entity (HA typically handles this gracefully with a warning log, but it is still sloppy). Could trigger unnecessary coordinator refreshes.

**Recommendation:** Add an early exit guard at key points in `_async_tick`:

```python
async def _async_tick(self, _now: datetime | None = None) -> None:
    """Periodic reconciliation tick."""
    if not self._started or not self._dirty:
        return
    self._dirty = False
    # ... rest of method, with additional guard after each await:
    if not self._started:
        return
```

This is defensive but prevents post-teardown work.

### 3. Key Limitation: PIN Values Are Write-Only

This is the critical design constraint for sync logic:

**GetCredentialStatus does NOT return the actual PIN value for standard credential types (PIN, RFID, biometric).** The `CredentialData` field in `GetCredentialStatusResponse` exists but is **only populated for ALIRO credentials** (which use public keys, not secrets). For PIN/RFID/biometric credentials, the field is null.

This means:
- You can **write** a PIN to the lock via `SetCredential`
- You can **check** if a credential slot is occupied via `GetCredentialStatus`
- You can **enumerate** which users have which credential slots via `GetUser`
- You **cannot read back** the actual PIN code that was set

**Implications for sync logic:**
- The integration must be the **source of truth** for PIN values, since the lock cannot be queried for them
- There is no way to detect if a PIN was changed out-of-band (for example, via the lock's keypad programming mode or another controller)
- Sync must be **write-forward**: push known PINs to the lock, rather than reading lock state and reconciling
- The best you can do is verify a slot is "occupied" and owned by the expected user, but you cannot verify the PIN content matches what you think it should be

### Summary for Provider-Agnostic Abstraction Design

The Matter credential model maps cleanly to a generic abstraction:
- **User** = slot-indexed entity with name, type, status, and credential rule
- **Credential** = slot-indexed entity within a credential type, holding opaque data, associated with exactly one user

## Type: `SlotCode(StrEnum)`

### Invariants Identified

1. **Two and only two sentinel states exist**: EMPTY (cleared) and UNKNOWN (occupied but unreadable)
2. **Sentinel values are distinct from actual PIN codes**: The string values "empty" and "unknown" serve as sentinel markers that should never collide with real PINs
3. **SlotCode values are immutable enum members**: Cannot be extended or modified at runtime
4. **Coordinator data maps slots to exactly one of three categories**: a readable string PIN, SlotCode.EMPTY, or SlotCode.UNKNOWN

### Ratings

- **Encapsulation**: 5/10
  `StrEnum` inherits from `str`, which means `SlotCode.EMPTY == "empty"` is `True`. This is a **leaky abstraction** -- the sentinel's string value is exposed as its identity. Any code doing `==` comparisons (rather than `is` identity checks) will conflate the sentinel with its backing string. The PR consistently uses `is` checks, which is correct, but this is an invariant enforced only by developer discipline, not by the type system.

- **Invariant Expression**: 6/10
  The union type `str | SlotCode` is clear about intent, and the docstring explains the semantics well. However, the type annotation itself does not prevent confusion: since `SlotCode` IS a `str` (via `StrEnum`), the union is technically redundant at runtime -- every `SlotCode` value already satisfies `str`. A type checker can distinguish them, but the type literally says "string or string-subclass," which is semantically weak. The real invariant is "either an opaque PIN string that is NOT a sentinel, or a sentinel enum member," and the type system cannot express this.

- **Invariant Usefulness**: 8/10
  This is a strong improvement over the previous `str | None` model. Replacing the overloaded empty string and None with explicit sentinels removes a meaningful class of bugs: empty string vs None vs "slot not in dict" ambiguity is eliminated. The UNKNOWN sentinel directly enables write-only lock support (Matter). The sync truth table in the design doc is clean and complete.

- **Invariant Enforcement**: 4/10
  There are several enforcement gaps:
  
  **StrEnum equality leakage**: If a user's PIN is literally `"empty"` or `"unknown"`, `pin == SlotCode.EMPTY` returns `True`. The `is` checks used throughout the PR are correct, but one slip to `==` introduces a bug. Line 261 in sync.py already has `lock_code == ""` (a leftover `==` check against the old empty string convention) sitting right next to the new `is` checks, demonstrating how easy it is to mix styles.
  
  **No construction-time validation**: Nothing prevents a provider from returning `SlotCode.EMPTY` when it means something else, or returning the plain string `"empty"` instead of the enum member. The coordinator has no validation on incoming data.
  
  **JSON round-trip erasure**: `json.dumps(SlotCode.EMPTY)` produces `"empty"`, and `json.loads(...)` produces a plain `str`. Any serialization/deserialization boundary silently loses the sentinel type. The websocket layer handles this explicitly, but it is a fragile boundary.

- **Eliminates None ambiguity**: The old model conflated "slot not present," "slot cleared," and "masked/unresolvable" into None and empty string. This PR cleanly separates all three.
- **Removes `_resolve_pin_if_masked`**: The most complex method in zwave_js.py is eliminated entirely. Resolution responsibility moves to the single consumer that needs it (the sensor), which is architecturally clean.
- **Clean sync truth table**: The `calculate_in_sync` logic becomes a straightforward pattern match with no hidden resolution steps. The UNKNOWN + ON = True semantic is well-reasoned and documented.
- **Design doc quality**: The design doc at `docs/plans/2026-04-02-slot-code-enum-design.md` thoroughly documents per-consumer behavior and rationale.
- **Consistent `is` usage**: The PR consistently uses identity checks (`is SlotCode.EMPTY`) rather than equality, which is the correct pattern for StrEnum sentinels.

### Concerns

1. **StrEnum is the wrong base class for sentinels.** The entire point of `SlotCode` is that its members are NOT regular strings -- they are sentinels that should be distinguishable from any PIN value. But `StrEnum` makes them `isinstance(SlotCode.EMPTY, str) == True` and `SlotCode.EMPTY == "empty" == True`. This directly undermines the core invariant. A plain `Enum` would make illegal comparisons fail loudly rather than silently succeed.

2. **Value collision with real PINs.** While "empty" and "unknown" are unlikely PINs on a numeric lock, nothing in the type system prevents this. If the project ever supports alphanumeric codes (RFID, etc., mentioned in the PR description), this becomes a real risk.

3. **Mixed comparison idioms in sync.py line 261.** The expression `lock_code is SlotCode.EMPTY or lock_code == ""` mixes the new `is`-based sentinel check with the old `==`-based empty string check. This suggests the migration is incomplete -- the `lock_code == ""` branch should either be removed (if providers are fully migrated) or documented as a transitional fallback. As written, it invites future developers to use `==` for SlotCode checks too.

4. **`SlotState.coordinator_code` is `str | SlotCode | None`.** This is a three-way union where `None` means "coordinator has no data for this slot." With `SlotCode.EMPTY` already representing "slot is cleared," the None case is narrow (only during startup before first poll). But having both None and SlotCode.EMPTY in the same field is the kind of ambiguity this PR is trying to eliminate. 5. **`isinstance(v, SlotCode)` as a negative check** (zwave_js.py line 592: `if not isinstance(v, SlotCode)`) is fragile. It means "has a readable PIN" but reads as "is not a SlotCode." If a third SlotCode member were added (say, `DISABLED`), this check would silently exclude it. A helper method like `is_readable_code(v)` would express intent more clearly.

1. **Switch from `StrEnum` to plain `Enum`.** The only reason to use `StrEnum` would be if you need the values to serialize transparently as strings. But the websocket layer already has explicit serialization logic for SlotCode, and JSON round-trips erase the type anyway. A plain `Enum` would cause `SlotCode.EMPTY == "empty"` to return `False`, making accidental equality comparisons fail loudly. If you need string representation for logging, `__str__` or `__repr__` overrides suffice.

   If `StrEnum` must be kept for some integration reason (for example, Home Assistant's data store expects string-compatible values), then at minimum add a comment on the class explaining why StrEnum was chosen and that `is` must always be used instead of `==`.

## Overall: `str | SlotCode` Union Design

The union `str | SlotCode` is the central type in this PR, and it has a fundamental tension: **`SlotCode` IS a `str` (via StrEnum), so the union is a lie at the runtime level.** The type checker treats them as distinct, but at runtime `isinstance(SlotCode.EMPTY, str)` is `True`. This means:

- You cannot use `isinstance(x, str)` to distinguish "readable PIN" from "sentinel" -- it returns True for both
- You must use `isinstance(x, SlotCode)` or `x is SlotCode.MEMBER` instead
- Any `==` comparison between a plain string and a SlotCode member that happens to have the same value will return True

The PR handles this correctly everywhere by using `is` checks, but the design carries ongoing maintenance risk. Every future contributor who touches coordinator data must know to use `is` not `==`. This is an invariant enforced by convention, not by the type system.

**If you keep StrEnum**, the recommendation is to pick sentinel string values that are impossible as real PINs. Currently `"empty"` and `"unknown"` are plausible alphanumeric codes. Values like `"__EMPTY__"` or `"\x00EMPTY"` would be safer, though uglier. Alternatively, switching to plain `Enum` eliminates the entire class of problems.

This is the only place in the PR that handles both representations. Other code paths (websocket, provider) only handle `SlotCode.EMPTY`.

If the code sensor's `native_value` returns `""` for `SlotCode.EMPTY` (which it does, line 91 of sensor.py), then `code_state` could be `""` when the coordinator had `SlotCode.EMPTY`. The fallback path (`coordinator_code is None`) would then see `""` and the `lock_code == ""` check would work. This is technically correct but fragile -- it relies on `sensor.py` translating `SlotCode.EMPTY` to `""` and the sync manager knowing to check for `""`.

**User Impact:** This dual representation is a maintenance hazard. A future developer might remove the `lock_code == ""` check thinking it is dead code (since the coordinator now uses `SlotCode.EMPTY`), breaking sync for the fallback path.

**Recommendation:** Add a comment explaining WHY both checks exist: the `SlotCode.EMPTY` check covers coordinator data, while the `""` check covers the sensor state fallback when coordinator data is unavailable. Better yet, consider normalizing `code_state` to `SlotCode` at the point where `SlotState` is constructed.
