**Confidence: 83**

**File:** `/Users/raman/projects/lock_code_manager/custom_components/lock_code_manager/sync.py`, lines 373–376 and lines 482–490


The post-sync re-check at lines 447–453 re-resolves state after the `async_refresh` completes. This works correctly for the in-flight sync path.

The gap is the following sequence:
1. Sync is in progress (lock held), coordinator update arrives → task is suppressed (line 484).
2. Sync completes, `_perform_sync` returns True.
3. `coordinator.async_refresh()` is called at line 433 — this fires coordinator listeners again. Since the lock is still held during the `await`, the new listener callback creates a task but the lock-check in `_async_check_and_sync` discards it.
4. The post-sync re-check at line 448 runs `_resolve_slot_state(None)` which reads the freshly-updated coordinator data directly. This re-check does catch the updated state.

```python
if not self._coordinator.last_update_success and not self._retry.active:
    return None
```

Since `self._retry.active` is False at that point (the retry was just scheduled, not running), this returns None even though a retry is queued. The slot will stay `in_sync=False` until the retry fires. This is correct behavior but the retry guard in `_resolve_slot_state` conflates "retry is scheduled" (`pending`) with "retry is executing" (`active`). The `OneShotRetry.active` property is only True during callback execution (see `util.py` line 146). The condition should be `not self._retry.pending` to cover the waiting-for-retry state, or `not (self._retry.pending or self._retry.active)`:

```python
if not self._coordinator.last_update_success and not self._retry.pending:
```

**Confidence: 88** — The `OneShotRetry.active` property is explicitly documented as True only during execution. Using it instead of `pending` here is a logic error that will cause updates to be dropped during the retry window.

### Issue 1: `_aio_lock.locked()` check in `_handle_coordinator_update` is not a reliable guard (Confidence: 88)

**File:** `/Users/raman/projects/lock_code_manager/custom_components/lock_code_manager/sync.py`, lines 506–516

In `_handle_coordinator_update`, the method checks `self._aio_lock.locked()` and returns early if locked. But because `_handle_coordinator_update` is a `@callback` (synchronous, runs on the event loop), and `async_create_task` just schedules a coroutine — it does not hold the lock at the time of the check. The lock is only held while `_async_check_and_sync` is actually running inside its `async with self._aio_lock` block.

The problem: if `_async_check_and_sync` is awaiting `_perform_sync` or `async_refresh()` (both of which are long-running awaits), the event loop can run other callbacks, including `_handle_coordinator_update`. At that point, `_aio_lock.locked()` returns `True` and `_handle_coordinator_update` returns early — dropping the coordinator update entirely without scheduling any retry. If `_async_check_and_sync` holds the lock through a coordinator refresh, and the coordinator fires an update while waiting, that update is silently dropped even though post-sync re-evaluation (`_resolve_slot_state(None)`) might now read stale state (before the fired update is reflected in `_coordinator.data`).

This is an inherent design tension: the coordinator fires `_handle_coordinator_update` precisely after refreshing its data, but if the lock is held during `await self._coordinator.async_refresh()`, the notification is dropped. The post-sync re-read at line 462 uses `_resolve_slot_state(None)` which reads `self._coordinator.data` directly — so it does capture the fresh data. This means the logic is actually correct in the common case: the refresh result is available immediately after `async_refresh()` returns, and the re-read at line 462 picks it up.

If the lock is held during the initial `_async_check_and_sync()` call in `async_start`, and a coordinator update arrives, the early return at line 506 drops it. **File:** `/Users/raman/projects/lock_code_manager/custom_components/lock_code_manager/sync.py`, lines 361–367

This catch-all is meant for unexpected errors, but if the error is actually permanent (e.g., a provider bug that raises the same exception on every call), the tick loop will retry forever at 5-second intervals: `_dirty=True` → tick fires → `_perform_sync` → exception → `_dirty=True` → tick fires → repeat.

### ISSUE 4: `async_stop` does not guard against a concurrently running `_async_tick`

**Location:** `/custom_components/lock_code_manager/sync.py`, `async_stop` method

**Severity:** MEDIUM

**Issue Description:**
`async_stop` is a synchronous `def` (not async), so it runs atomically within a single event loop iteration. It sets `self._started = False`, cancels the tick unsub, cleans up listeners, and resets the tracker. However, if `_async_tick` is currently mid-execution (sitting at an `await` point like `_perform_sync` or `coordinator.async_refresh()`), the tick will resume after that await and continue operating on a stopped manager.

Specifically, after `async_stop`:
- `_async_tick` could call `self._write_state()` on an entity that has been removed
- `_async_tick` could call `self._coordinator.async_refresh()` after the coordinator unsub
- `_async_tick` could set `self._dirty = True`, though this is harmless

The `_tick_unsub()` cancels future scheduled ticks, but does NOT cancel an already-running `_async_tick` coroutine.

**User Impact:** Could cause `async_write_ha_state` on a removed entity (HA typically handles this gracefully with a warning log, but it is still sloppy). Could trigger unnecessary coordinator refreshes.

**Recommendation:** Add an early exit guard at key points in `_async_tick`:

```python
async def _async_tick(self, _now: datetime | None = None) -> None:
    """Periodic reconciliation tick."""
    if not self._started or not self._dirty:
        return
    self._dirty = False
    # ... rest of method, with additional guard after each await:
    if not self._started:
        return
```

This is defensive but prevents post-teardown work.
