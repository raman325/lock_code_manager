# Error Handling Audit: PR #987 `fix/repair-issue-lifecycle`

## Summary

This PR adds repair issue lifecycle management: deleting stale `slot_disabled_`, `pin_required_`, and `lock_offline_` issues at appropriate points. It also changes the switch `async_turn_on` to raise `HomeAssistantError` instead of silently returning, and wraps `_disable_slot` in try/except/finally. The changes are generally well-structured with good test coverage. I found issues ranging from CRITICAL to MEDIUM.

## Issue 1: Bare `except Exception` in `_disable_slot` catches too broadly

**Location:** `/Users/raman/projects/lock_code_manager/custom_components/lock_code_manager/sync.py`, lines 315-319

**Severity:** HIGH

**Issue Description:** The `except Exception` block in `_disable_slot` catches every possible exception from `async_disable_slot`, including errors that have nothing to do with the disable operation itself. The `async_disable_slot` function calls `hass.services.async_call` and `async_create_issue` internally. This catch block will silently absorb:

- `KeyError` / `AttributeError` from programming bugs in `async_disable_slot` or its callers
- `HomeAssistantError` from the service layer (e.g., service not registered, entity not available)
- `TypeError` from incorrect argument passing
- Any corruption in the entity registry lookup

**Hidden Errors:** The `_LOGGER.exception` does log the full traceback, which is good. However, the method continues to `_reset_sync_tracker()` in the `finally` block regardless. This means the circuit breaker state resets even when the slot was never actually disabled, which could lead to the circuit breaker re-triggering and attempting to disable the slot again in an infinite loop of failed disable attempts that all get caught and swallowed.

**User Impact:** If `async_disable_slot` consistently fails (e.g., the switch entity was removed), the user sees no repair issue, the sync tracker resets, and the circuit breaker fires again next window, creating an endless log-spam cycle of `"Failed to disable slot"` exception messages with no user-facing notification.

**Recommendation:** At minimum, create the repair issue directly in the `except` block as a fallback, so the user at least knows something went wrong. Consider also re-raising after logging if the exception is not a transient service call failure.

```python
async def _disable_slot(self, reason: str) -> None:
    """Disable the slot and create a repair issue."""
    try:
        await async_disable_slot(
            self._hass,
            self._ent_reg,
            self._config_entry.entry_id,
            self._slot_num,
            reason=reason,
            lock_name=self._lock.display_name,
            lock_entity_id=self._lock.lock.entity_id,
        )
    except Exception:
        _LOGGER.exception(
            "%s: Failed to disable slot, resetting sync tracker anyway",
            self._log_prefix,
        )
        # Still create the repair issue so the user is notified
        async_create_issue(
            self._hass,
            DOMAIN,
            f"slot_disabled_{self._config_entry.entry_id}_{self._slot_num}",
            is_fixable=True,
            is_persistent=True,
            severity=IssueSeverity.WARNING,
            translation_key="slot_disabled",
            translation_placeholders={
                "slot_num": str(self._slot_num),
                "reason": f"{reason} (Note: automatic disable may have failed — check slot state manually)",
            },
        )
    finally:
        self._reset_sync_tracker()
```

## Issue 3: `get_entry_data(config_entry, CONF_LOCKS, [])` returns lock entity IDs from config, not runtime locks

**Location:** `/Users/raman/projects/lock_code_manager/custom_components/lock_code_manager/__init__.py`, line 471
