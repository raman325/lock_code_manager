**Confidence: 83**

**File:** `/Users/raman/projects/lock_code_manager/custom_components/lock_code_manager/sync.py`, lines 373–376 and lines 482–490


The post-sync re-check at lines 447–453 re-resolves state after the `async_refresh` completes. This works correctly for the in-flight sync path.

The gap is the following sequence:
1. Sync is in progress (lock held), coordinator update arrives → task is suppressed (line 484).
2. Sync completes, `_perform_sync` returns True.
3. `coordinator.async_refresh()` is called at line 433 — this fires coordinator listeners again. Since the lock is still held during the `await`, the new listener callback creates a task but the lock-check in `_async_check_and_sync` discards it.
4. The post-sync re-check at line 448 runs `_resolve_slot_state(None)` which reads the freshly-updated coordinator data directly. This re-check does catch the updated state.

```python
if not self._coordinator.last_update_success and not self._retry.active:
    return None
```

Since `self._retry.active` is False at that point (the retry was just scheduled, not running), this returns None even though a retry is queued. The slot will stay `in_sync=False` until the retry fires. This is correct behavior but the retry guard in `_resolve_slot_state` conflates "retry is scheduled" (`pending`) with "retry is executing" (`active`). The `OneShotRetry.active` property is only True during callback execution (see `util.py` line 146). The condition should be `not self._retry.pending` to cover the waiting-for-retry state, or `not (self._retry.pending or self._retry.active)`:

```python
if not self._coordinator.last_update_success and not self._retry.pending:
```

**Confidence: 88** — The `OneShotRetry.active` property is explicitly documented as True only during execution. Using it instead of `pending` here is a logic error that will cause updates to be dropped during the retry window.

### Issue 1: `_aio_lock.locked()` check in `_handle_coordinator_update` is not a reliable guard (Confidence: 88)

**File:** `/Users/raman/projects/lock_code_manager/custom_components/lock_code_manager/sync.py`, lines 506–516

In `_handle_coordinator_update`, the method checks `self._aio_lock.locked()` and returns early if locked. But because `_handle_coordinator_update` is a `@callback` (synchronous, runs on the event loop), and `async_create_task` just schedules a coroutine — it does not hold the lock at the time of the check. The lock is only held while `_async_check_and_sync` is actually running inside its `async with self._aio_lock` block.

The problem: if `_async_check_and_sync` is awaiting `_perform_sync` or `async_refresh()` (both of which are long-running awaits), the event loop can run other callbacks, including `_handle_coordinator_update`. At that point, `_aio_lock.locked()` returns `True` and `_handle_coordinator_update` returns early — dropping the coordinator update entirely without scheduling any retry. This is the correct intent per the comment "a sync is in progress and will read latest state when it runs", but the problem is the re-read at lines 461–467 only runs for the `not self._in_sync` case. If `_async_check_and_sync` holds the lock through a coordinator refresh, and the coordinator fires an update while waiting, that update is silently dropped even though post-sync re-evaluation (`_resolve_slot_state(None)`) might now read stale state (before the fired update is reflected in `_coordinator.data`).

This is an inherent design tension: the coordinator fires `_handle_coordinator_update` precisely after refreshing its data, but if the lock is held during `await self._coordinator.async_refresh()`, the notification is dropped. The post-sync re-read at line 462 uses `_resolve_slot_state(None)` which reads `self._coordinator.data` directly — so it does capture the fresh data. This means the logic is actually correct in the common case: the refresh result is available immediately after `async_refresh()` returns, and the re-read at line 462 picks it up.

However, there is a subtler gap: `_handle_coordinator_update` also handles the case where `_in_sync is None` (initial sync trigger). If the lock is held during the initial `_async_check_and_sync()` call in `async_start`, and a coordinator update arrives, the early return at line 506 drops it. But `async_start` is `async_create_task`'d, and since `_aio_lock` is acquired inside `_async_check_and_sync`, the coordinator listener fires after setup — it is plausible for `_handle_coordinator_update` to be called while `_aio_lock` is held during initial sync. If the initial `_async_check_and_sync` returns early (e.g., entities not ready at line 234), `_in_sync` stays `None` but the lock is released. The coordinator update is only dropped if the lock is actively held. So the early-return at line 506 is fine — it's guarding against re-entrant scheduling, not against missing updates.

On reflection, the `_aio_lock.locked()` check in `_handle_coordinator_update` does provide meaningful protection: it prevents queuing a new task when the current `_async_check_and_sync` is mid-flight and will re-read state via `_resolve_slot_state(None)` post-sync. The design is coherent. I'm withdrawing this as a high-confidence issue.

### Issue 3: `_sync_attempts_exceeded` checks the wrong comparison direction (Confidence: 90)

**File:** `/Users/raman/projects/lock_code_manager/custom_components/lock_code_manager/sync.py`, lines 361–367

```python
def _sync_attempts_exceeded(self) -> bool:
    if self._sync_attempt_count < MAX_SYNC_ATTEMPTS:
        return False
    if self._sync_attempt_first is None:
        return False
    return dt_util.utcnow() - self._sync_attempt_first <= SYNC_ATTEMPT_WINDOW
```

`MAX_SYNC_ATTEMPTS = 3`. This returns `True` (exceeded) only when `count >= 3` AND the window hasn't expired.

But look at where `_record_sync_attempt()` is called (line 289) — it is called on **every set operation**, regardless of whether the set succeeded from the lock's perspective. After 3 set calls within the window, `_sync_attempts_exceeded()` returns True and the slot is disabled.

The logic is: if we've tried 3 times and we're still not in sync, something is wrong. That intent is clear.

Now check the interaction: `_record_sync_attempt()` is called inside `_perform_sync` **before** confirming the lock actually accepted the code (line 289 — it's called right after `await self._lock.async_internal_set_usercode(...)` but before the coordinator refresh that would confirm the code took). This means a transient network error that raises `LockDisconnected` would not call `_record_sync_attempt` (since the exception is caught first), but a silent failure (no exception, code sent but lock ignores it) would increment the counter correctly. The design seems intentional — track how many times we've **sent** a set command without achieving sync.

However, there is a real logical issue: `_record_sync_attempt` resets the count at the top if the window has expired (lines 351–355), then increments. After reset, `_sync_attempt_first` is set to `now`. So after 3 attempts within the window, the circuit breaks. But consider:

- Attempt 1: count=1, first=T0
- Attempt 2: count=2
- Attempt 3: count=3, `_sync_attempts_exceeded` is called in the **next** `_async_check_and_sync` invocation (line 422). But `_sync_attempts_exceeded` is checked **before** `_perform_sync` is called for attempt 4. So: after 3 recorded attempts, on the 4th trigger of `_async_check_and_sync`, `_sync_attempts_exceeded()` returns True and disables the slot.

This means `MAX_SYNC_ATTEMPTS = 3` actually allows 3 set operations, then disables on the 4th trigger. The naming is semantically slightly off (it's "exceeded after 3 attempts" = "4th attempt is blocked"), but the behavior is correct per the design intent documented in the comment. This is not a bug.

I'm satisfied this is working as intended.

### Issue 5: `calculate_in_sync` falls back to `code_state` when `coordinator_code is None`, but `coordinator_code` is `None` only when the slot is absent from coordinator data (Confidence: 90)

**File:** `/Users/raman/projects/lock_code_manager/custom_components/lock_code_manager/sync.py`, lines 256–270

```python
if self._in_sync is None and int(self._slot_num) not in self._coordinator.data:

if int(self._slot_num) not in self._coordinator.data:
```
